Operating Systems (OS) Core
========================================

1. OPERATING SYSTEM ROLES & ARCHITECTURE
----------------------------------------
An Operating System is system software that manages hardware resources, files, memory, and devices, providing common services for application programs.
- Kernel: The core component of the OS with direct access to hardware.
  * Monolithic Kernel: Entire OS services run in kernel space (e.g., Linux, Windows). Fast but a single crash takes down system.
  * Microkernel: Only core services (IPC, memory) run in kernel space; others (drivers, filesystems) run in user space (e.g., Mach, L4). More stable but higher IPC overhead.
- Dual Mode Execution: User Mode (limited privileges) vs. Kernel Mode (full privileges). Mode transition is triggered by System Calls or Hardware Interrupts.

2. PROCESS MANAGEMENT & CPU SCHEDULING
----------------------------------------
- Process: Program in execution. Contains text (code), data (global variables), heap (dynamic memory), and stack (local variables).
- Threads: Lightweight processes. Share code, data, and OS resources, but have private stacks and registers.
- Process Control Block (PCB): Data structure storing process info (state, PC, registers, memory limits, files).
- Context Switching: Saving state of active process and loading state of next process. CPU overhead.
- Scheduling Algorithms:
  * Preemptive vs Non-Preemptive.
  * FCFS (First-Come-First-Serve): Non-preemptive. Suffers from Convoy Effect.
  * SJF (Shortest Job First): Optimal average waiting time. SJF preemptive is Shortest Remaining Time First (SRTF).
  * Round Robin (RR): Preemptive. Each process gets a time slice (quantum). Good for time-sharing.
  * Priority Scheduling: Starvation of low priority processes is solved by Aging.

3. PROCESS SYNCHRONIZATION
----------------------------------------
- Race Condition: Multiple processes write/read same data concurrently, outcome depends on execution order.
- Critical Section: Code block where shared resources are accessed. Requires Mutual Exclusion, Progress, and Bounded Waiting.
- Synchronization Tools:
  * Mutex: Mutual exclusion lock. Binary state (locked/unlocked).
  * Semaphores: Integer variables accessed via atomic operations `wait()` (P) and `signal()` (V).
    - Counting Semaphore: Unlimited domain.
    - Binary Semaphore: Maps between 0 and 1.
- Classical Problems: Producer-Consumer, Readers-Writers, Dining Philosophers.

4. DEADLOCKS
----------------------------------------
A state where a set of processes are blocked because each holds a resource and waits for another resource held by another process.
- Coffman Conditions (All 4 must hold):
  1. Mutual Exclusion: At least one resource must be non-shareable.
  2. Hold and Wait: Process holds a resource while waiting for another.
  3. No Preemption: Resources cannot be forcibly taken.
  4. Circular Wait: Process A waits for B, B waits for C, C waits for A.
- Handling Deadlocks:
  * Prevention: Structure constraints to deny one of the 4 conditions.
  * Avoidance: Use dynamic checks. Bankers Algorithm calculates safe states using max claims.
  * Detection and Recovery: Allow deadlocks, detect via Resource Allocation Graphs, recover by terminating processes or preempting resources.
  * Ignorance: Pretend deadlocks never occur (Ostrich Algorithm - used by Linux/Windows).

5. MEMORY MANAGEMENT
----------------------------------------
- Logical vs. Physical Address: Logical generated by CPU; physical is actual memory location. Mapped by MMU.
- Contiguous Memory Allocation: Partitions memory. Suffers from Fragmentation:
  * Internal Fragmentation: Allocated memory is slightly larger than requested.
  * External Fragmentation: Total free memory is sufficient but not contiguous. Solved by Compaction.
- Paging: Non-contiguous allocation. Divides logical memory into Pages and physical memory into Frames. Uses Page Tables. No external fragmentation, but internal fragmentation can exist in last page.
- Segmentation: Divides memory into logical segments (code, stack, heap).

6. VIRTUAL MEMORY & PAGE REPLACEMENT
----------------------------------------
Virtual memory allows execution of processes not completely in physical memory, using disk swapping.
- Page Fault: CPU accesses page not loaded in physical memory. Triggers disk read.
- Thrashing: OS spends more time swapping pages in/out than executing processes.
- Page Replacement Algorithms:
  * FIFO (First-In-First-Out): Suffers from Belady's Anomaly (page faults increase as physical memory frames increase).
  * LRU (Least Recently Used): Replaces page unaccessed for longest time. Optimal approximation.
  * Optimal Page Replacement: Replaces page unaccessed for longest future time (theoretical benchmark).
